# Объяснение по заданию

## Общая идея

В задании отрабатывается практическая сборка и запуск приложения через Docker. Есть готовое FastAPI-приложение, MySQL и два прокси: Nginx принимает входящий HTTP-запрос, HAProxy передает его в приложение, приложение пишет запись в базу.

Цепочка получается такая:

```text
curl/browser -> Nginx:8090 -> HAProxy:8080 -> FastAPI:5000 -> MySQL
```

## Почему нужен `Dockerfile.python`

Исходный `Dockerfile` в проекте был заготовкой. Для задания нужен отдельный `Dockerfile.python`.

Я сделал multistage-сборку:

1. Первый stage `builder` собирает Python wheels из `requirements.txt`.
2. Второй stage ставит готовые wheels и копирует приложение.

Так итоговый образ остается простым, а установка зависимостей отделена от runtime-части.

## Зачем `.dockerignore`

`.dockerignore` нужен, чтобы в Docker build context не улетали лишние файлы: `.git`, кеши Python, виртуальные окружения, логи, дампы и прочее. Это ускоряет сборку и уменьшает шанс случайно положить мусор в образ.

## Что делает `compose.yaml`

`compose.yaml` подключает готовый `proxy.yaml` через `include`, как требует задание. В нем добавлены два сервиса:

- `web` - FastAPI-приложение, собирается из `Dockerfile.python`;
- `db` - MySQL 8.0.

Для `web` передаются переменные подключения к базе:

- `DB_HOST=db`;
- `DB_USER`;
- `DB_PASSWORD`;
- `DB_NAME`.

Для `db` используются переменные из `.env`, чтобы не писать пароль прямо в compose-файл.

## Почему MySQL 8.0, а не просто latest

Тег `mysql:8` сейчас может подтянуть свежую ветку MySQL, где изменена поддержка старых плагинов авторизации. Для backup через образ `schnitzler/mysqldump` это ломает подключение.

Поэтому использован `mysql:8.0`: это все еще MySQL 8, но он нормально работает с `mysql_native_password`, который понимает используемый образ для дампа.

## Как проверялась работа приложения

После запуска compose выполнялась команда:

```bash
curl -L http://127.0.0.1:8090
```

Важно ходить именно на `8090`, потому что это порт Nginx. Если идти напрямую на `5000`, приложение покажет предупреждение, что запрос идет не через правильную прокси-цепочку.

После HTTP-запроса приложение создает таблицу `requests` и пишет туда время запроса и IP.

Проверка в MySQL:

```sql
show databases;
use virtd;
show tables;
SELECT * from requests LIMIT 10;
```

## Что делает `deploy.sh`

Скрипт нужен для облачной VM или любой чистой Linux VM.

Режим по умолчанию:

```bash
sudo ./deploy.sh
```

Он проверяет Docker, при необходимости устанавливает его, клонирует fork в `/opt/shvirtd-example-python` и запускает compose.

Отдельный режим backup:

```bash
sudo ./deploy.sh backup
```

Он читает `.env`, находит compose-сеть проекта и запускает контейнер `schnitzler/mysqldump`, который делает дамп MySQL в `/opt/backup`.

## Почему пароль не записан в backup-команду явно

В скрипте пароль не прописан строкой. Он берется из `.env`:

```bash
source "${app_dir}/.env"
```

Это лучше, чем хранить пароль в cron или отдельной команде. В реальном проекте `.env` тоже не коммитят, но в учебном репозитории он уже был частью задания.

## Зачем cron

Cron показывает, что backup можно запускать регулярно:

```cron
* * * * * /opt/shvirtd-example-python/deploy.sh backup >/var/log/shvirtd-backup.log 2>&1
```

В задании просили раз в минуту, поэтому использовано `* * * * *`.

## Что происходит в задаче с Terraform

Там идея такая: Docker-образ - это набор слоев. Внутри образа `hashicorp/terraform:latest` есть бинарник `/bin/terraform`.

Первый способ:

1. Скачать образ.
2. Посмотреть образ через `dive`.
3. Сделать `docker save` в tar-архив.
4. Найти слой, где лежит `/bin/terraform`.
5. Достать файл и запустить `terraform version`.

Второй способ проще:

1. Создать контейнер из образа.
2. Выполнить `docker cp container:/bin/terraform ./terraform`.
3. Удалить временный контейнер.
4. Проверить `terraform version`.

## Как запускалось в Yandex Cloud

На Yandex Cloud VM был поднят обычный Ubuntu-хост. На него через SSH был склонирован fork проекта и запущен `deploy.sh`.

Скрипт поставил Docker, собрал образ приложения и поднял compose-проект в `/opt/shvirtd-example-python`.

Проверка выполнялась по адресу:

```text
http://89.169.179.78:8090
```

Через check-host должен пройти путь:

```text
Internet -> Nginx -> HAProxy -> FastAPI -> MySQL
```

После внешнего запроса в таблице `virtd.requests` появилась запись с IP клиента. Backup на этой же VM проверен вручную через `deploy.sh backup`, затем добавлен запуск через cron раз в минуту.
