# Пояснение по заданию

## Что тут вообще отрабатывалось

Docker Swarm нужен, чтобы запускать контейнеры не на одной машине, а на группе машин. У Swarm есть manager-нода, которая управляет кластером, и worker-ноды, которые выполняют задачи.

В этом задании смысл был такой:

1. Поднять несколько VM в одной сети.
2. Поставить Docker на каждую.
3. Объединить их в один Swarm-кластер.
4. Проверить, что manager видит worker-ноды.
5. По желанию запустить приложение как Swarm services.

То есть это уже не просто `docker run`, а первый шаг к нормальному запуску сервиса на нескольких серверах.

## Manager и worker

Manager - это управляющая нода. На ней выполняются команды типа:

```bash
docker node ls
docker service ls
docker service create
```

Worker - это обычная рабочая нода. Она подключается к manager через:

```bash
docker swarm join ...
```

Join-token в отчет не пишется. Это как временный ключ подключения к кластеру.

## Почему использовался private IP

Swarm инициализировался командой:

```bash
docker swarm init --advertise-addr 10.129.0.7
```

`10.129.0.7` - приватный IP manager внутри Yandex Cloud. Так правильнее, потому что ноды находятся в одной внутренней сети и им не нужно общаться друг с другом через публичные адреса.

## Что такое service

В обычном Docker мы запускаем контейнер:

```bash
docker run nginx
```

В Swarm запускается service:

```bash
docker service create --name web nginx
```

Service - это описание нужного состояния. Например: "хочу один контейнер nginx". Swarm сам решает, где его запустить, и следит, чтобы он был живой.

Если контейнер упал, Swarm пробует поднять его заново.

## Overlay network

Для связи сервисов в Swarm создается overlay-сеть:

```bash
docker network create --driver overlay shvirtd-net
```

Она нужна, чтобы сервисы могли общаться по именам, даже если контейнеры окажутся на разных нодах.

Например, приложение может ходить к базе по имени `shvirtd-db`, а не по IP.

## Почему с приложением был нюанс

В предыдущем задании приложение запускалось через Docker Compose. Там есть `build`, то есть Compose сам собирает image из Dockerfile.

В Swarm через `docker stack deploy` так обычно не делают. Swarm ожидает уже готовый image, который можно скачать из registry.

Поэтому был учебный вариант:

1. Собрал image `shvirtd-web:swarm` на manager.
2. Запустил MySQL как service.
3. Запустил web как service.
4. Ограничил web manager-нодой:

```bash
--constraint node.role==manager
```

Это сделано потому, что image лежал только на manager. Если бы Swarm попытался запустить web на worker, worker не нашел бы image.

В нормальном варианте надо так:

```bash
docker build -t ghcr.io/user/app:tag .
docker push ghcr.io/user/app:tag
docker service create ghcr.io/user/app:tag
```

Тогда любая нода сможет скачать образ и запустить контейнер.

## Routing mesh и порт 8090

Сервис был опубликован так:

```bash
--publish published=8090,target=5000
```

Это значит:

- снаружи открывается порт `8090`;
- внутри контейнера приложение слушает `5000`.

В Swarm есть routing mesh: опубликованный порт может принять запрос на ноде кластера и довести его до нужного контейнера.

## Почему стенд удален

Три VM в облаке стоят денег, даже если нагрузка маленькая. Поэтому после проверки они удалены.

Для учебы это нормальная схема: поднял стенд, снял выводы, удалил стенд.

## Что было важно понять

Swarm - это не про "один контейнер на одной машине". Это про сервисы, которые могут жить на нескольких нодах.

Главные команды:

```bash
docker swarm init
docker swarm join
docker node ls
docker service create
docker service ls
docker service ps
```

Главная практическая мысль: если сервис должен запускаться на разных нодах, образ должен лежать в registry, а секреты и пароли не надо писать в отчетах и репозитории.
