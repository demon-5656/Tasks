# Объяснение по заданию

Это задание про базовую работу с Docker и Compose.

## Что тут вообще отрабатывали

Главная идея: не просто скачать nginx, а пройти обычный набор действий:

1. Собрать свой образ.
2. Запустить контейнер с нужным именем и портом.
3. Проверить, что порт реально слушается.
4. Понять, что происходит при `docker attach`.
5. Посмотреть разницу между портом внутри контейнера и проброшенным портом Docker.
6. Подключить одну папку в разные контейнеры.
7. Запустить несколько сервисов через Compose.
8. Поднять локальный registry.
9. Задеплоить контейнер через Portainer.

То есть задание простое, но полезное: много мелких Docker-команд, на которых обычно и спотыкаются.

## Почему сломался curl после смены порта nginx

Контейнер был запущен так:

```bash
-p 127.0.0.1:8080:80
```

Это значит:

```text
хост 127.0.0.1:8080 -> контейнер 80
```

Потом внутри контейнера nginx поменяли с `listen 80` на `listen 81`.

Docker об этом не знает. Он все еще отправляет трафик на порт `80` внутри контейнера. А там уже никто не слушает.

Поэтому внутри контейнера `curl 127.0.0.1:81` работает, а с хоста `curl 127.0.0.1:8080` нет.

## Зачем volume в задаче 4

Volume тут показывает, что контейнеры могут видеть общий каталог хоста.

Если один контейнер создал файл в `/data`, второй контейнер тоже его видит, потому что на самом деле это один и тот же каталог на VM.

Это часто используют для конфигов, логов, данных и обмена файлами.

## Что показал Compose

Если рядом лежат `compose.yaml` и `docker-compose.yaml`, Docker Compose выбирает `compose.yaml`.

Чтобы использовать оба файла, в новом Compose можно подключить один файл из другого через:

```yaml
include:
  - docker-compose.yaml
```

После этого одним `docker compose up -d` поднялись и Portainer, и registry.

## Зачем локальный registry

Registry нужен, чтобы хранить Docker-образы не в DockerHub, а локально.

В задании образ `custom-nginx` был загружен сюда:

```text
127.0.0.1:5000/custom-nginx
```

Потом Portainer взял этот образ и запустил stack.

## Что важно запомнить

- `docker build` собирает образ.
- `docker run -d` запускает контейнер в фоне.
- `docker rename` переименовывает контейнер.
- `docker attach` цепляется к основному процессу контейнера.
- `docker exec` запускает команду внутри контейнера.
- `docker rm -f` удаляет даже работающий контейнер.
- `-v host_path:container_path` подключает папку хоста в контейнер.
- `docker compose up -d` поднимает проект.
- `docker compose down` гасит compose-проект.
- Проброс портов Docker не меняется сам, если внутри контейнера поменять конфиг nginx.
