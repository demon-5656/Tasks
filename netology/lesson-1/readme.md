# Домашнее задание к занятию 1. Введение в виртуализацию

## Задача 1

Практическая часть предполагает создание экономичной ВМ в Yandex Cloud на базе образа `Yandex Cloud Toolbox`, проверку SSH-доступа и установленного Docker.

Команда для проверки Docker на ВМ:

```bash
sudo docker --version
```

Согласно странице образа `Yandex Cloud Toolbox` в Yandex Cloud Marketplace, скрипт `setup.sh` настраивает рабочее окружение для `kubectl`, Docker, YC CLI и Terraform, а также меняет Bash Command Prompt. Если в настройках ВМ указан сервисный аккаунт, для YC CLI автоматически создается профиль `default`.

Источник: https://yandex.cloud/ru/marketplace/products/yc/toolbox

Для экономии ресурсов ВМ следует создавать как прерываемую, с минимально достаточной конфигурацией CPU/RAM и публичным IP только на время выполнения задания. После завершения работы ВМ нужно удалить либо оставить для следующих заданий, если она действительно будет использоваться.

## Задача 2

### Высоконагруженная база данных MySQL, критичная к отказу

Оптимальный вариант: **физические серверы**.

Критерии выбора:

- минимальные накладные расходы на виртуализацию;
- максимальная производительность дисковой подсистемы и сети;
- предсказуемая задержка ввода-вывода;
- проще гарантировать ресурсы под конкретную критичную БД.

Для отказоустойчивости при этом нужны репликация, кластеризация, резервное копирование, мониторинг и отдельная архитектура хранения данных. Виртуализация возможна, но для действительно нагруженной и критичной БД физические серверы обычно дают больше контроля над производительностью.

### Различные web-приложения

Оптимальный вариант: **виртуализация уровня ОС**.

Критерии выбора:

- web-приложения удобно запускать в контейнерах;
- быстрый старт и остановка экземпляров;
- простое масштабирование;
- экономное использование ресурсов;
- удобная доставка приложений через Docker-образы.

Контейнеризация хорошо подходит для stateless-сервисов, микросервисов, frontend/backend-приложений и окружений, где нужно быстро разворачивать одинаковые экземпляры.

### Windows-системы для бухгалтерского отдела

Оптимальный вариант: **паравиртуализация**.

Критерии выбора:

- Windows требует полноценной изоляции гостевой ОС;
- можно централизованно управлять ВМ;
- проще делать резервные копии и снапшоты;
- удобно выделять ресурсы под пользователей или отделы;
- хорошая совместимость с корпоративной инфраструктурой.

Контейнеризация уровня ОС здесь не подходит, потому что бухгалтерские Windows-приложения обычно требуют полноценной Windows-среды, пользовательского интерфейса, доменной интеграции и совместимости с периферией.

### Системы для высокопроизводительных расчётов на GPU

Оптимальный вариант: **физические серверы**.

Критерии выбора:

- максимальная производительность GPU без лишних накладных расходов;
- прямой доступ к видеокартам;
- меньше проблем с драйверами и passthrough;
- проще использовать CUDA, ROCm и специализированные HPC-библиотеки.

Виртуализация с GPU passthrough возможна, но она усложняет настройку и сопровождение. Для высокопроизводительных вычислений физические серверы обычно надежнее и эффективнее.

## Задача 3

### Сценарий 1

Условия: 100 ВМ Linux и Windows, общие задачи, преимущественно Windows-инфраструктура, нужны программные балансировщики, репликация данных и автоматизированное резервное копирование.

Выбор: **Microsoft Hyper-V / Windows Server с System Center или Windows Admin Center**.

Обоснование:

- хорошая интеграция с Windows-инфраструктурой и Active Directory;
- нормальная поддержка Windows и Linux ВМ;
- есть механизмы репликации, резервного копирования и централизованного управления;
- подходит для инфраструктуры, где основная экспертиза связана с Microsoft.

### Сценарий 2

Условия: нужно наиболее производительное бесплатное open source-решение для небольшой инфраструктуры 20-30 серверов Linux и Windows.

Выбор: **KVM с Proxmox VE**.

Обоснование:

- KVM является производительным гипервизором Linux;
- Proxmox VE дает удобную web-панель управления;
- поддерживаются Linux и Windows ВМ;
- есть кластеры, снапшоты, резервное копирование, миграция ВМ;
- решение бесплатное и open source.

### Сценарий 3

Условия: нужно бесплатное, максимально совместимое и производительное решение для виртуализации Windows-инфраструктуры.

Выбор: **Microsoft Hyper-V Server / Hyper-V роль в Windows Server**.

Обоснование:

- лучшая совместимость с Windows-гостями;
- поддержка драйверов и интеграционных служб Microsoft;
- хорошая производительность для Windows-нагрузок;
- удобная интеграция с доменом, групповыми политиками и Windows-инструментами администрирования.

Если требуется строго бесплатная платформа без покупки Windows Server, можно рассматривать бесплатные варианты Hyper-V, но в реальной инфраструктуре все равно нужно учитывать лицензирование гостевых Windows-систем.

### Сценарий 4

Условия: нужно рабочее окружение для тестирования продукта на нескольких дистрибутивах Linux.

Выбор: **VirtualBox или VMware Workstation Player**.

Обоснование:

- удобно запускать несколько тестовых ВМ на рабочей станции;
- легко создавать и откатывать снапшоты;
- можно быстро проверять продукт на разных дистрибутивах;
- не требуется полноценный серверный кластер виртуализации.

Если тестирование нужно автоматизировать, дополнительно можно использовать Vagrant, который позволяет описывать тестовые окружения как код.

## Задача 4

Гетерогенная среда виртуализации - это инфраструктура, где одновременно используются несколько платформ виртуализации, например VMware, Hyper-V, KVM, Proxmox или облачные гипервизоры.

Возможные проблемы:

- сложнее администрировать инфраструктуру из-за разных интерфейсов и подходов;
- нужны специалисты с компетенциями сразу по нескольким платформам;
- усложняется мониторинг и сбор метрик;
- разные форматы виртуальных дисков и шаблонов ВМ;
- сложнее переносить ВМ между платформами;
- различаются механизмы снапшотов, резервного копирования и репликации;
- выше риск ошибок при эксплуатации;
- сложнее обеспечить единые политики информационной безопасности;
- может увеличиться стоимость поддержки и лицензирования;
- труднее автоматизировать процессы создания, миграции и восстановления ВМ.

Что нужно сделать для минимизации рисков:

- стандартизировать основные платформы и использовать их только там, где это оправдано;
- вести единую документацию по архитектуре и регламентам;
- внедрить централизованный мониторинг;
- использовать единые политики резервного копирования;
- описывать инфраструктуру как код, например через Terraform или Ansible;
- унифицировать сетевые схемы, правила доступа и требования безопасности;
- регулярно проверять восстановление из резервных копий;
- обучать администраторов работе со всеми используемыми платформами;
- заранее описать правила миграции ВМ между средами.

Если бы у меня был выбор, я бы не создавал гетерогенную среду без необходимости. Лучше выбрать одну основную платформу виртуализации и использовать ее как стандарт. Гетерогенная среда оправдана только тогда, когда есть сильные технические или организационные причины: разные требования отделов, миграция со старой платформы, зависимость от конкретного ПО, требования законодательства или необходимость использовать облако вместе с локальной инфраструктурой.
